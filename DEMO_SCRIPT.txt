[1:00 — ARCHITECTURE]

(show assets/architecture_diagram.png)

Let me walk you through how the system works under the hood.

Every patient message first hits the input_classifier node.
This normalises the input and stores it in shared workflow state.

Next, the security_checkpoint node runs automatically on every query.
It scrubs PII — things like Pakistani phone numbers, CNIC identity numbers,
and email addresses. It also scans for prompt injection attacks.
Any suspicious keyword like "ignore instructions" or "jailbreak" is
flagged as a CRITICAL security event and rerouted immediately.

After security passes, the orchestrator routes the message to one of
three specialised sub-agents.

FAQ questions go to the faq_node, which uses our MCP server tools —
specifically get_service_catalog and check_availability — to answer
precisely with real clinic prices and slot data.

Booking requests go to the booking_flow node. This uses rerun_on_resume
to safely collect all required fields: patient name, phone number,
service selection, preferred date, and time slot. The MCP tools
check_availability, create_booking, and get_booking_details are wired in.

When all booking details are collected, the workflow pauses at the
human_approval node — that is our Human-in-the-Loop checkpoint.
A receptionist reviews and confirms before the booking is finalised.

Complex cases and security events go to the escalation_node, which
empathises, logs a reference ticket, and provides emergency contact details.

All paths converge at final_output, which formats the response based
on the outcome and delivers it back to the patient.

[3:00 — SECURITY & MCP]

The security checkpoint runs on every single message — no exceptions.
It produces a structured JSON audit log with fields for PII scrubbing status,
injection detection, classification intent, and severity level.
This gives clinic administrators a full, searchable audit trail.

The MCP Server exposes five tools specific to this dental domain:
get_service_catalog, check_availability, create_booking,
get_booking_details, and cancel_booking.
These are wired into both the FAQ agent and the booking agent,
so every response is grounded in real operational data.

And finally — the WhatsApp webhook. The FastAPI server includes
GET and POST endpoints that integrate directly with the
Meta WhatsApp Cloud API. Every incoming patient message maps to
a unique ADK session, runs through the full agent workflow,
and the response is sent back automatically — zero human touchpoints required.
